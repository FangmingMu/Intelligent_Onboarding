# [安全] 内部核心系统权限申请审批矩阵 (2026 Q2 修订版)

## 0. 权限管理通用守则 (General Policy)
1. **最小权限原则 (Least Privilege)**: 员工仅能申请当前工作必需的最低权限。
2. **时效性原则**: 所有短期项目权限（Temp-Access）默认有效期为 30 个自然日，到期自动回收（Auto-Revoke）。
3. **安全审计 (Security Audit)**: 权限申请行为将被记录于 `Security-Access-Log`，并由公司安全委员会（Sec-Com）定期抽检。

## 1. 系统分级定义 (System Tiers)
- **P1 (核心生产系统)**: 生产环境数据库（Prod-DB）、生产 Kubernetes 控制平面（Prod-K8s-Master）、支付网关（Pay-Gateway）。
- **P2 (研发基础设施)**: 代码仓库（GitLab）、持续集成（CI/CD-Runner）、容器镜像仓库（Harbor）。
- **P3 (日常办公系统)**: 企业邮箱（Exchange）、飞书/钉钉、Jira/Confluence、VPN。

## 2. 权限申请审批矩阵 (Approval Matrix)
根据系统重要性，必须满足以下硬性逻辑条件。

| 目标系统 (Target System) | 申请权限类型 | 必需岗位等级 (Required Title) | 强制审批流 (Mandatory Workflow) | 风险等级 (Risk Level) |
| :--- | :--- | :--- | :--- | :--- |
| **P1: 生产环境数据库** | 写权限 (Write) | P7 (高级专家) 及以上 | 申请人 -> 部门总监 (Director) -> DBA Team -> CTO 报备 | **CRITICAL** (高危) |
| **P1: 生产环境数据库** | 读权限 (Read) | P6 (高级研发) 及以上 | 申请人 -> 部门经理 (Manager) -> DBA Team | **HIGH** (高) |
| **P2: GitLab 代码库** | 管理员 (Admin) | P8 (总监) 及以上 | 申请人 -> 技术保障中心 (SRE-Team) -> 安全委员会 | **HIGH** (高) |
| **P2: CI/CD 管道** | 编辑 (Editor) | 无限制 (P1-P10) | 申请人 -> 组长 (Lead) | **MEDIUM** (中) |
| **P3: VPN 远程访问** | 全局访问 (Global) | 无限制 (P1-P10) | 申请人 -> 直属上级 | **LOW** (低) |

## 3. 强制性逻辑判定规则 (Harness Rules)
Agent 在解析用户意图并执行 Tool Calling 前，必须严格检查以下逻辑：

1. **[R-101] 身份校验失效判定**: 
    - 若申请【P1 核心生产系统】读写权限，而申请人 Title 为【初级研发 (P1-P3)】或【中级研发 (P4-P5)】，系统必须**立即驳回**，严禁触发审批流。
    - *判定逻辑提示*: `If System_Tier == 'P1' and User_Title < 'P6' then Action = REJECT`.

2. **[R-102] 审批链路不全判定**:
    - 凡涉及【高危操作】（如删除生产库、重置管理员账号），申请描述（Description）必须包含“项目代号”与“业务影响分析（BIA）”。
    - 若工单未包含【部门总监】审批节点，视为**流程缺陷**，Agent 应挂起任务并反问用户。

3. **[R-103] 多因子二次确认**:
    - 在执行诸如“重置 VPN 密码”、“解封账户”等动作前，Agent 必须调用 `Send-MFA-Code` 接口向用户手机发送验证码，并要求用户在对话框输入回显。

4. **[R-104] 高压线规则 (Red-Line)**:
    - 禁止通过 Agent 申请他人（Other User）的非公开信息权限（如薪资单、人事档案）。
    - 任何违反此项的行为将自动触发 `Security-Alert-S1` 并抄送安全合规部。

## 4. 常见问题 (FAQ)
Q: “为什么我的 P1 权限申请被驳回了？”
A: 检查您的职级是否满足 P6/P7 的硬性要求。如果是紧急故障排查（Emergency Fix），请先联系 SRE-Team 获取临时账号（Temp-Account）。

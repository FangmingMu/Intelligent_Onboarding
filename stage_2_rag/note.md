# Stage 2: 语义搜索与 RAG 引擎 (RAG Engine) —— 面试核心知识点

本层级负责“知识检索”，让 Agent 能够基于私有文档回答问题。以下是面试中关于 RAG 的高频考点：

## 1. 语义切分 (Semantic Chunking)
*   **知识点**：为什么不用 `RecursiveCharacterTextSplitter`（按长度切），而用 `MarkdownHeaderTextSplitter`？
*   **理解**：
    *   **上下文保留**：按长度切可能会把一个段落甚至一个表格切断。
    *   **语义完整性**：按 Markdown 的标题（H1, H2, H3）切分，能确保每个片段都是一个独立的逻辑块（如一个完整的 VPN 配置步骤）。
    *   **元数据注入**：切分时自动把标题信息存入 Metadata，检索时能让模型知道这个片段属于哪个章节。

## 2. 向量嵌入 (Embedding) 与向量数据库 (Vector DB)
*   **知识点**：什么是 Embedding？为什么用 FAISS？
*   **理解**：
    *   **Embedding**：将文字转化为一串数字（向量）。语义相近的句子，在多维空间里的距离也相近。
    *   **FAISS (Facebook AI Similarity Search)**：
        *   它不是传统的数据库，而是一个**高性能索引库**。
        *   它利用了特定的算法（如 IVFFlat）来极速计算向量之间的距离。
*   **面试加分项**：你可以提到我们选择了 Qwen3-Embedding，因为它在中文语义理解上表现优异。

## 3. 检索算法：相似度计算
*   **知识点**：系统是怎么找到相关片段的？
*   **理解**：通过计算用户问题的向量与数据库中片段向量的 **余弦相似度 (Cosine Similarity)** 或 **点积 (Dot Product)**。
*   **区别**：余弦相似度更关注方向（语义），点积更关注强度。在 NLP 中，余弦相似度是最常用的衡量标准。

## 4. 两阶段检索：召回 (Recall) + 重排 (Rerank) —— 【简历核心亮点】
*   **知识点**：为什么要 Rerank？
*   **理解**：
    *   **召回 (Recall)**：从几万个片段里快速找回最像的 10 个（速度快，但不够精准）。
    *   **重排 (Rerank)**：用更高精度的模型（BGE-Reranker）对这 10 个片段进行精细打分排序（精度极高，但计算开销大，所以只对前 10 个做）。
*   **价值**：显著解决了 RAG 系统中“搜得到但排在后面的干扰项多”的问题，极大提升了回答的准确率。

## 5. 提示词工程 (RAG Prompt)
*   **核心逻辑**：`已知信息 + 用户问题 + 约束指令`。
*   **约束的作用**：防止“幻觉”。我们在 Prompt 中明确要求：“如果已知信息不足，请直说不知道”。

## 附录：核心概念深度解析

### 1. 什么是“幻觉 (Hallucination)”？RAG 怎么解决它？
*   大模型有时会一本正经地胡说八道。RAG 通过给模型提供“参考资料”，让模型从“闭卷考试”转为“开卷考试”，从而强制其回答必须基于事实。

### 2. 为什么不用 Fine-tuning (微调) 而是用 RAG？
*   **时效性**：RAG 只要更新文档，回答立刻变；微调需要重新训练模型，慢且贵。
*   **准确性**：RAG 可以提供来源出处（溯源）；微调后的知识会混在模型权重里，无法溯源。
*   **数据权限**：RAG 可以根据用户身份检索不同的文档；微调模型无法实现权限控制。

### 4. 向量数据库选型对比：FAISS vs. Chroma vs. Milvus
*   **FAISS (本项目采用)**：
    *   *定位*：纯索引库，不具备数据库的管理功能（如 CRUD 审计、用户管理）。
    *   *优点*：极轻量、极快，适合本地部署和小规模知识库。
*   **ChromaDB**：
    *   *定位*：面向 AI 开发者的向量数据库。
    *   *优点*：内置了 Embedding 模型管理，上手极快，支持简单的持久化。
*   **Milvus (生产级标配)**：
    *   *定位*：分布式向量数据库。
    *   *优点*：支持海量数据（亿级）、高可用集群、云原生架构。面试时可以说：“本项目采用 FAISS 保证本地演示性能，在实际生产中可平替为 Milvus 以支持大规模并发。”

### 5. 数学底层：余弦相似度 vs. 点积
*   **点积 (Dot Product)**：计算两个向量每一位对应相乘再求和。
    *   *直观理解*：它不仅看方向，还看“长度”。如果一个片段里关键词出现的次数非常多，它的向量长度就大，点积就高。
*   **余弦相似度 (Cosine Similarity)**：计算两个向量夹角的余弦值。
    *   *直观理解*：它**只看方向**，不看长度。它衡量的是“语义的相似度”而非“关键词的堆砌程度”。在 NLP 中，由于文档长度不一，余弦相似度通常更稳健。

### 7. 向量数据库选型：真的只是规模区别吗？
*   **不只是规模，更是“工程化程度”的区别**：
    *   **FAISS (库)**：像一个高性能的**计算引擎**。它只负责算，不负责存。你得自己写代码处理数据的加载、保存和元数据关联。
    *   **Chroma (轻量数据库)**：像一个**单机版数据库**。它把计算和存储打包好了，适合单机运行的小项目。
    *   **Milvus (企业级系统)**：像一个**完整的分布式集群**。它支持：
        *   **多租户隔离**：不同业务数据互不干扰。
        *   **高可用**：一台服务器挂了，检索不中断。
        *   **标量过滤**：支持复杂的 SQL 语句与向量搜索混用。
        *   **数据持久化与备份**：有完善的快照和恢复机制。

### 9. 检索策略对比实验 (k=7 深度测试)
为了探究不同检索策略对系统表现的影响，我们在 k=7（初始召回数量）的条件下进行了对比测试：

| 指标 | 纯向量检索 (Vector) | 混合检索 (Hybrid) | 差异 (Delta) |
| :--- | :--- | :--- | :--- |
| **Faithfulness** | 0.648 | 0.632 | -1.6% |
| **Answer Relevancy** | 0.702 | 0.692 | -1.0% |
| **Context Recall** | **0.875** | **0.875** | **持平** |
| **Context Precision** | 0.808 | **0.858** | **+5.0%** |

#### 实验洞察与面试复盘话术：
1.  **为什么 Recall（召回率）持平？**：在小规模私有知识库中，正确答案往往在 Top-K 范围内已经能够被两种算法共同覆盖，因此召回上限趋同。
2.  **为什么 Hybrid 的 Precision（精确度）更高？**：这是本次实验最重要的发现。由于企业文档包含大量强特征词（如 `vpn-gw-shanghai.corp-sec.com`），**BM25 算法起到了“语义锚点”的作用**。它能比向量检索更早、更准地锁定原句，从而在 Rerank 之前就为重排器提供更高质量的候选列表，最终提升了最相关片段的排位。
3.  **结论**：Hybrid Search 虽然在召回上限上与向量持平，但其**检索结果的稳定性与精准排位能力**显著优于单一模式，是更适合生产环境的选择。

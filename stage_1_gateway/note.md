# Stage 1: 网关与路由层 (Gateway & Routing Layer) —— 面试核心知识点

本层级是 Agent 系统的“前额叶”，负责感知用户意图并决定任务调度路径。以下是您需要掌握的核心技术概念：

## 1. 意图识别与分发 (Intent Routing)
*   **知识点**：为什么不用传统的正则或 BERT 分类模型，而用大模型做路由？
*   **理解**：
    *   **灵活性**：LLM 能够理解复杂的自然语言语义，不需要针对每个新意图训练专门的分类器。
    *   **冷启动**：只需通过 Prompt 描述规则，即可实现高精度的分类。
*   **代码体现**：`router.py` 中的 `destination` 分流逻辑。

## 2. 结构化输出 (Structured Output) —— 【高频考点】
*   **知识点**：如何保证大模型返回的总是你想要的 JSON，而不是一堆废话？
*   **技术细节**：我们使用了 LangChain 的 `with_structured_output` 结合 Pydantic 模型。
*   **底层原理**：
    *   它通常利用了 OpenAI 兼容接口的 **Function Calling** 或 **JSON Mode** 特性。
    *   通过将 Pydantic Schema 转换为 JSON Schema 发给模型，强制模型按格式填空。
*   **意义**：这是 Agent 走向“工程化”的基石，确保了下游代码调用的稳定性。

## 3. 对话上下文管理 (Context Awareness)
*   **知识点**：如何解决“我是张三”这种指代不明的问题？
*   **理解**：
    *   **上下文丢失问题**：单轮对话无法识别依赖前文的意图。
    *   **窗口记忆 (Window Memory)**：我们通过在 `app.py` 中维护 `st.session_state.messages`，并在路由时只截取最近 3-5 轮对话注入 Prompt。
    *   **权衡**：既保留了必要的背景信息，又避免了 Token 浪费和因历史过长导致的“模型幻觉”。

## 4. 架构设计：为什么需要 Gateway 层？
*   **解耦**：将意图识别与具体任务（RAG/Action）分离，方便独立迭代。
*   **性能**：对于简单的闲聊，不需要去检索向量库，节省了大量耗时和算力成本。
*   **安全性**：可以在网关层统一拦截非法指令或敏感词。

## 5. Streamlit 响应式框架
*   **知识点**：Streamlit 的运行机制。
*   **理解**：每次页面交互（如点击按钮、输入文字），Streamlit 都会**从头到尾重新运行一次脚本**。
*   **Session State**：由于脚本是重运行的，所以必须使用 `st.session_state` 来持久化存储聊天记录，否则每轮对话都会被重置。

## 附录：核心技术深度解析

### 1. 传统的正则/BERT 分类 vs. LLM 路由
*   **正则 (Regex)**：基于关键词匹配（如：只要有“密码”就走重置流程）。
    *   *缺点*：太死板，无法处理“我把那个登入的东西忘了”这种没有关键词但语义明确的句子。
*   **BERT 分类模型**：一种深度学习模型（如文本分类器）。
    *   *缺点*：需要几千条标注好的数据进行训练，增加一个新功能（如“小秘书闲聊”）就得重新训练并部署一个新模型，维护成本极高。
*   **LLM 路由 (本项目采用)**：利用大模型的语义理解力。只需在 Prompt 里写几句说明，它就能识别极其含蓄或复杂的意图，实现“零样本 (Zero-shot)”分类。

### 2. `destination` 分流逻辑的具体过程
1.  **定义规则**：在 `router.py` 中，我们给模型下达指令（System Prompt），明确告知三种路径的定义。
2.  **格式约束**：通过 Pydantic 定义一个只能包含 `RAG`, `ACTION`, `CHAT` 的枚举列表。
3.  **模型决策**：LLM 阅读用户的问题，结合规则，在脑中匹配最合适的标签。
4.  **代码跳转**：`app.py` 拿到字符串标签后，通过 `if...elif...` 语句像调度员一样切换后台代码块。

### 3. Pydantic Schema 是什么？
*   **通俗理解**：它是 Python 的**“数据合同”**。
*   **作用**：你定义一个类（Class），规定某个字段必须是字符串、某个字段必须在特定范围内。
*   **在 Agent 中的意义**：LLM 本质上是生成概率最大的下一个字，它是不可控的。Pydantic 就像一个**模具**，强制将 LLM 喷涌而出的文字“挤”进固定的 JSON 格式里。如果 LLM 生成了不符合格式的内容，Pydantic 会立刻报错拦截，防止后端代码崩溃。

### 4. Streamlit 的 `st.session_state` 详解
*   **`messages` 是固定的吗？**：不是。`st.session_state` 本质上是一个**空的字典 (Dictionary)**。`messages` 是我们自己起的名字，你也可以叫它 `chat_history`。
*   **存储在哪里？**：存储在**服务器的内存 (RAM)** 中。它与每个用户的浏览器标签页（Session）一一对应。
*   **是持久存储吗？**：**不是**。
    *   如果你刷新页面或关闭标签页，内存就会被清空，聊天记录随之消失。
    *   这就是为什么在 Stage 4 中我们需要把日志写入 `logs.jsonl`（磁盘存储），那才是真正的“持久化”。
*   **为什么要用它？**：因为 Streamlit 每次交互都会从头运行脚本，如果没有这个内存字典，脚本运行到第二行时，第一行的变量就会被重置。`st.session_state` 是 Streamlit 唯一能让变量“跨越刷新”活下来的地方。
